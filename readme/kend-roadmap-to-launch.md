## 병렬 개발 전략 (Phase 1~2 핵심)

> TossPayments 결제와 주문/배송 도메인은 DB 설계만 맞춰두면 병렬 개발 가능

```
[결제 플로우]                    [주문/배송 플로우]
결제창 연동                      주문 상태 머신 설계
  ↓                               ↓
결제 성공 → orders(status: paid) ← DB 공유
  ↓                               ↓
결제 취소/환불                   배송 처리 (Seller)
                                  ↓
                               구매확정 → 정산
```

결제 완료 시 `orders` 테이블에 `status: 'paid'`만 기록하면  
이후 배송/정산은 그 위에 독립적으로 쌓이는 구조.

## Phase 1 — 핵심 기반 완성 + 병렬 착수 (5~6월)

### 1-1. 휴대폰 인증 도입 ★최우선★

> ⚠️ **휴대폰 인증(SMS OTP) ≠ NICE 본인확인**
> - **SMS OTP**: 알리고/NHN Cloud 기반. 사업자 없이 개발/테스트 가능. 아이디찾기·비번찾기·소셜 추가정보에 사용.
> - **NICE 본인확인**: 법인 + 계약 필요. 결제 실운영 시 본인확인 목적. 별도 항목으로 분리.

**예상 공수**: 8~12일 | **난이도**: 높음

| 세부 작업 | 공수 |
|----------|------|
| SMS OTP 발송 연동 (알리고 또는 NHN Cloud) | 2일 |
| 휴대폰 인증 UI (인증번호 입력, 재발송, 타이머) | 2일 |
| 아이디 찾기 flow | 1~2일 |
| 비밀번호 찾기 flow | 1~2일 |
| 소셜 회원가입 추가정보 입력 flow | 2일 |
